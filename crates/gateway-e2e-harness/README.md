# gateway-e2e-harness

Thin orchestrator: reads **curl templates** (`$VAR` placeholders) from `cases/*.json`, loads `.env`,
runs `curl`, and writes **pretty-printed JSON** (`serde_json::to_string_pretty`) to the output directory.

## Environment variables (curl templates)

| Variable | Description |
| ------ | ------ |
| `GATEWAY_BASE` | Gateway base URL, e.g. `http://127.0.0.1:8080` |
| `ALEPHANT_CONTROL_PLANE_API_KEY` | **Required** for auth-bearing cases: expands `Authorization: Bearer $ALEPHANT_CONTROL_PLANE_API_KEY` in curl templates; should match the gateway `.env` (locally may be the same test key as a VK) |
| `HARNESS_E2E_RUN_TOKEN` | **Optional**: when unset, the harness fills this from the current `--run-id` (or default timestamp `…-e2e`). Cases like `KV-004` embed it in the prompt so repeated full runs do not keep hitting stale cache and fail `notContains: HIT`. Set in `.env` or via `--var HARNESS_E2E_RUN_TOKEN=…` for reproducibility. |
| `MULTI_MODEL_MOCK_E2E_ENABLE` | **Optional**: when set to any non-empty value, `tc-mm-*.json` cases run (otherwise skipped per `requiredEnv`). For sibling dir `multi_model_service`: run `python main.py seed` on Postgres, start the mock HTTP server, ensure mock `master_keys.base_url` matches the mock listen address, then use seeder-generated **`legend_{provider_code}`** as Bearer (see that dir’s `README.md`). |

  ```bash
  python3 crates/gateway-e2e-harness/scripts/sync_multi_model_mock_cases.py
  ```

- **Hand-authored extras**: `tc-mm-002-openai-chat-stream-mock.json` (streaming), `tc-mm-007` … `tc-mm-010` (`/v1/embeddings`, `/v1/completions`, `/v1/responses`, `/v1/images/generations`).

### `requiredEnv` (optional)

Case JSON may declare top-level **`requiredEnv`**, a string array of env var names that must exist and be non-blank before the case runs, for example:

```json
"requiredEnv": ["OPENAI_API_KEY", "EMBEDDING_MODEL_NAME"]
```

If any listed variable is missing or blank, the harness marks the case **skipped** (not a failure) and does not run `curl` / `curlPrelude` / `ch`. Matching rows in `e2e-report.json` have `skipped: true` and `record.skipped.missingEnv` lists missing vars; executed cases have `skipped: false`.

### `profile: ["manual"]`

These cases are placeholders or need **special setups** (auth disabled, rate limits saturated, separate compat gateway, etc.) and **do not run** under default `gate` / `full` (see `case_matches_profile` in `case.rs`). Extend this crate’s CLI or copy JSON and change `profile` when you need them.

## Unified API conventions (current cases)

- **Path**: `POST $GATEWAY_BASE/ai/chat/completions` (matches multi-model docs).
- **`model`**: Must be **`{provider}/{model_id}`** (e.g. `openai/gpt-4o-mini`, `anthropic/claude-3-haiku`). Bare model names do not satisfy Unified parsing.
- If **`compat_mode`** is enabled, Unified may pin to the OpenAI vendor, conflicting with `provider/model` routing; confirm compat is off before `full` (per real `config`).

Cases use **`Authorization: Bearer $ALEPHANT_CONTROL_PLANE_API_KEY`** in JSON; set that variable in `.env` or the environment before running (aligned with control-plane auth; often a valid virtual key).

## `curlPrelude` (optional)

- Type: array of strings; each entry matches main **`curl`**—a curl-command template starting with `curl` (`$VAR` supported).
- Behavior: each prelude runs **once in order** **before** the main `curl`; output is discarded (side effects only, e.g. filling a “1 req/s per IP” window).
- Assertions: still apply only to the **last** main `curl` response (same as a lone `curl`).

Examples for **global client-IP rate limiting**: `cases/tc-iprl-001-second-health-429.json`, `cases/tc-iprl-002-x-forwarded-for-429.json` (`profile: ["manual"]`, requires gateway `global.client-ip-rate-limit` with `requests-per-second: 1`; IPRL-002 also needs trusted proxy CIDRs including `127.0.0.0/8` so local `curl` can trust `X-Forwarded-For`).

## Hand-case tiers (`TIER: A-strict` / `TIER: C-probe`)

Matches [design spec](../../docs/superpowers/specs/2026-04-27-gateway-e2e-harness-semantic-prompt-cases-design.md):

| Tier | `intent` prefix | Assertions |
| ---- | ---------------- | ------ |
| **A-strict** | `TIER: A-strict; REQUIRES: ...` | Usually `httpStatus: 200` plus stable `body`/`headers`/`json`; confirm REQUIRES deps before running |
| **C-probe** | `TIER: C-probe; may not 200; ...` | Often `httpStatus: { "in": [ 200, 400, ... ] }`; non-200 is not a harness failure |

- **`$HARNESS_E2E_RUN_TOKEN`:** For cache-related A-strict, same idea as `KV-004`—token isolates prompts across runs (see `HARNESS_E2E_RUN_TOKEN` above).
- **A-strict `curlPrelude`:** If **prelude HTTP failure must fail the whole case**, add **`curl -f`** to prelude and main `curl` (`curl` exits non-zero on 4xx/5xx; see `curl_run`).

**Hand-authored filename prefixes** `tc-sc-` (semantic), `tc-pm-` (prompt/params) are **not** generated from the matrix by `generate_tc_matrix.py`; the script’s `files` dict **must not** contain conflicting keys. Re-running the matrix generator **does not** delete or overwrite `tc-*.json` files on disk that are **not** listed in `files`; still do not add those prefix keys to `files`.

**LCS-011 (large-context truncate A-strict) deferred**: `openai/gpt-4o-mini` has a ~**128k** token window in the gateway; a single inline `curl` `messages[0].content` at ordinary sizes **rarely** exceeds the raised input budget reliably to trigger `Alephant-Token-Limit-Exception-Handler: truncate`. If we later add **file-backed** huge bodies or a smaller-context stand-in model, we can add `LCS-011` A-strict.

### Semantic-cache hand cases

`SC-001` / `SC-002` send semantic-cache headers:

- `Alephant-Embeddings-Key: $OPENAI_API_KEY`
- `Alephant-Embeddings-Model: $EMBEDDING_MODEL_NAME`

Both declare `requiredEnv: ["OPENAI_API_KEY", "EMBEDDING_MODEL_NAME"]`; missing either skips per `requiredEnv` instead of reporting missing embeddings as a business failure.

`SC-001` is C-probe: checks semantic-cache headers and Unified `/ai/chat/completions` are accepted; no HIT required. `SC-002` is a strict HIT probe under seeded Qdrant/cache: `curlPrelude` warms cache; main request uses a paraphrased prompt and asserts `alephant-cache: HIT`. Note: `alephant-cache: HIT` alone proves a cache hit, not uniquely semantic vs exact KV.

### Default-model manual cases

`DM-001` and `DM-002` omit `model` in the body and use `profile: ["manual"]`; the runner must prepare DB / policy seed first.

- `DM-001` positive probe: expects seeded policy with an allowed chat-like default (`chat` / `multimodal` / `reasoning`); gateway injects default model and returns success.
- `DM-002` negative probe: run only when seed explicitly creates **no** chat-like candidates; exercises existing no-candidate / no-model error paths.

## Assertions `assertions` (optional)

Beyond `httpStatus` and `headers`:

| Block | Role |
| ---- | ------ |
| `body` | `contains`: substring list (all required); `notContains`: forbidden substrings; `regex`: single Rust `regex` match on full body. |
| `json` | After parsing body as JSON: `pathExists` (JSON Pointer list); `pathEquals` (pointer → expected); `arrayMinLength` (pointer → min length). Pointer syntax: [`serde_json::Value::pointer`](https://docs.rs/serde_json/latest/serde_json/struct.Value.html#method.pointer). |
| `sse` | On **streaming** full response: `dataFramesMin`: minimum `data:` lines; `hasDone`: require `[DONE]` (OpenAI streaming). |

Successful Unified chat shapes live in the generator as `OPENAI_CHAT_OK_JSON`; errors as `API_ERROR_JSON` (`/error`, `/error/message`).

## ClickHouse (optional `ch` block)

| Variable | Description |
| ------ | ------ |
| `CLICKHOUSE_HTTP_URL` or `CLICKHOUSE_URL` | e.g. `http://127.0.0.1:8123` |
| `CLICKHOUSE_DATABASE` | Optional |
| `CLICKHOUSE_USER` / `CLICKHOUSE_PASSWORD` | Optional Basic auth |

SQL may use **`$ALEPHANT_ID`** from the prior response header `alephant-id`.

### `ch.poll` and `ch.assertPoll`

| Field | Role |
| ------ | ------ |
| `poll` (optional, default `maxAttempts` 30, `backoffMs` 500) | Retries only **`clickhouse_query` `run_sql` failures** (network / HTTP) with backoff; **does not** keep waiting when the query succeeds but returns no rows / row assert fails. |
| `assertPoll` (optional, same defaults) | Applies only when each **`clickhouse_query` is immediately followed by `assert_row`** in `steps` (no `sleep` or other steps between): after a successful query + parsed rows, if `assert_row` fails, the harness **re-runs the same expanded SQL** after `backoffMs` until success or `maxAttempts`. Without `assertPoll`: single query + single assert. If `assertPoll` is set but pairing is invalid, the case fails upfront with a clear reason. |

Failure prefixes: `ch: query:` (query retries exhausted), `ch: assert:` (single assert failed without `assertPoll`), `ch: assertPoll:` (poll assert exhausted; message includes last row sample).

Minimal sketch (real CH + tables required):

```json
"ch": {
  "assertPoll": {},
  "steps": [
    { "type": "sleep", "ms": 200 },
    { "type": "clickhouse_query", "sql": "SELECT … WHERE id = '$ALEPHANT_ID' FORMAT JSONEachRow" },
    { "type": "assert_row", "match": { "request_id": "…" } }
  ]
}
```

`OBS-009` is `profile: ["manual"]` minimal `ch.assertPoll` plumbing with `requiredEnv: ["CLICKHOUSE_HTTP_URL"]`. It runs `SELECT 1 AS ok FORMAT JSONEachRow` to prove `clickhouse_query` + `assert_row` polling works; it does not validate gateway log schema; missing `CLICKHOUSE_HTTP_URL` skips per `requiredEnv`.

## Default `gate`: one failure per category

`--profile gate` (CLI default) runs only cases tagged `gate`. To avoid duplicate failure classes in CI, **each negative/probe category keeps one representative case on gate**; siblings run under `full` (or use `profile` only `full` in the matrix).

Current gate mapping (`caseId`):

| Category | Gate representative |
| ------ | ----------- |
| Liveness / health | `CFG-001` |
| Missing credentials | `ERR-001` |
| Invalid Bearer | `l1-auth-invalid-bearer-unified` |
| Unknown `provider/model` (Unsupported model) | `ERR-003` |
| Wrong HTTP method/path | `ERR-004` |
| Unknown global path (404) | `ROUTE-005` |

Duplicates (`AUTH-002` vs `ERR-001` both missing Authorization; `l1-health-*` vs `CFG-001` both `/health`) no longer repeat on gate; use `--profile full` for full coverage.

Running `full` **without** providers/models configured yields many `Unsupported model` / `Unsupported provider` responses—that is **expected**, not a harness defect.

## Gate vs full assumptions

- **`--profile gate`**: Does not assume all providers / real models; checks process reachability (`CFG-001`), basic auth/route negatives, **ERR-003** (unknown `provider/model`), etc.
- **`--profile full`**: Assumes plan-required providers, models, or equivalents for success paths and deep coverage.

Under `gate`, only **`CFG-001`** (`/health`) asserts **`httpStatus: 200`**; `l1-health-get` / `l1-health-head` repeat under `full` only.

## Commands

```bash
# Gate: only cases with `gate` profile (default)
cargo run -p gateway-e2e-harness -- --profile gate

# Cases needing real OpenAI VK (`profile: ["full"]`)
cargo run -p gateway-e2e-harness -- --profile full

# Custom .env and output dir
cargo run -p gateway-e2e-harness -- --dotenv-path ./.env --out-dir ./tmp/e2e-out --run-id local1
```

Output directory (**both files written every run**):

- **`e2e-report.json`**: Top-level `schema_version`, `run_id`, `profile`, `generated_at`, `cases`. `cases` lists **passing cases only**; each has `case_id`, `passed`, `failure_reason` (`null` on success), `record`.
- **`e2e-failures.json`**: Same shape as `e2e-report.json`; `cases` lists **failed cases only** (`passed` false, `failure_reason` string). Empty `[]` when none.
- On failure, stderr still prints `FAIL <id>: <reason>` per case.
- If any request returns **HTTP 400**, stderr also prints the **full response** (`status` / `timeTotalSec` / all headers / `body`).

Cases skipped via `requiredEnv` appear in `e2e-report.json` with `passed: true`, `skipped: true`, `failure_reason: null`, and `record.skipped.missingEnv`; executed cases have `skipped: false`.

## Case `profile`

- Omitted or `[]`: run under every `--profile`.
- `["gate"]`: only `--profile gate`.
- `["full"]`: only `--profile full`.
- `["gate","full"]`: both.
- `["manual"]` (this repo’s `tc-*`): **neither profile**, see above.
